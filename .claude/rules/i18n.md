# Internationalization (i18n)

## All User-Facing Strings Use t()

Every string visible to the user must go through `react-i18next`'s `t()` function. No hardcoded display text in JSX.

```tsx
// Wrong
<h1>Welcome back</h1>;

// Correct
const { t } = useTranslation();
<h1>{t("dashboard.welcomeBack")}</h1>;
```

## Translation Key Format

Keys follow **dot notation** scoped by feature and section:

```
feature.section.key
```

Examples:

```
auth.login.title
auth.login.submitButton
profile.settings.changePassword
common.actions.save
common.actions.cancel
```

## Supported Locales

The project supports:

- `en` — English
- `es` — Spanish

Both locale files must contain every key. Missing keys in either locale is a build-time error.

## Hook Usage

Use the `useTranslation()` hook in components:

```tsx
import { useTranslation } from "react-i18next";

export function LoginPage() {
  const { t } = useTranslation();
  return <h1>{t("auth.login.title")}</h1>;
}
```

## Locale File Location

Translation files live in:

```
src/shared/infrastructure/i18n/locales/
  en.json
  es.json
```

## What Does Not Need i18n

- Internal log messages and error codes.
- Developer-facing strings (console output, comments).
- Test fixture data.
